Stage 2 — Abstraction

Three independent decision procedures each read a subset of the NormalisedFinding and produce a rated value plus a trace: the branches taken and the fields read. No text is produced here — only structured outcomes that the next layer will narrate.

{# ── Tree 1: Exploitation likelihood ─────────────────────────────────── #}

Tree 1 — Exploitation likelihood → {{ ll }}

Asks: given what is publicly known about this vulnerability and asset, how likely is it that an attacker can exploit it? Reads exploit_available, known_exploited, and asset_internet_facing.

    {% for step in explanation.evidence_pointers.exploitation_likelihood.path %}
  1. {{ step }}
    2. {% endfor %}
{% if ll == "Unknown" %}
Why Unknown? exploit_available is listed in absent_by_format for {{ finding.source_format }}. The tree detected a missing required field and returned Unknown immediately without reading any further field. This is a structural limit of the source format, not a data quality problem.
{% elif ll == "High" and el_inputs.known_exploited %}
Why High? Both exploit_available = True and known_exploited = True. A working exploit exists and it is already being used against real targets. The tree stops here — internet-facing status is not needed to reach High when active exploitation is confirmed.
{% elif ll == "High" and el_inputs.asset_internet_facing %}
Why High? A public exploit is available and the asset is internet-facing. Even without confirmed active exploitation, an attacker can reach the vulnerable service directly from the internet, making successful exploitation highly likely.
{% elif ll == "Medium" and el_inputs.exploit_available %}
Why Medium? A public exploit is available, but asset_internet_facing = False and no active exploitation is confirmed. An attacker would need internal network access to reach the service, which reduces — but does not eliminate — the risk.
{% elif ll == "Medium" %}
Why Medium? Exploitation has been observed in the wild despite no public exploit being known. This unusual combination still warrants a Medium rating because real-world harm has occurred.
{% elif ll == "Low" %}
Why Low? No public exploit exists and no active exploitation has been reported. Both conditions must be absent for the tree to reach the lowest-risk outcome.
{% endif %}

Fields read: {{ el_inputs }}

{# ── Tree 2: Business impact ──────────────────────────────────────────── #}

Tree 2 — Business impact → {{ bi }}

Asks: if this vulnerability were exploited, how severely would it affect the business? Combines the asset environment (production vs non-production) and the technical severity band (from CVSS or scanner category) to enter an impact matrix. When a CMDB record was found during ingestion, a second elevated matrix becomes reachable.

    {% for step in explanation.evidence_pointers.business_impact.path %}
  1. {{ step }}
    2. {% endfor %}
{% if bi == "Unknown" %}
Why Unknown? Either asset_environment or the technical severity band was missing. Without both axes the impact matrix cannot be entered.
{% elif finding.asset_business_critical is none %}
Why preliminary? asset_business_critical is None — no CMDB asset record matched the asset identifier during ingestion. The tree took the preliminary path, which uses the standard matrix and caps impact at High. The system never infers business criticality from scanner data alone.
{% elif finding.asset_business_critical or finding.asset_owner_privileged %}
Why complete (elevated)? A CMDB record was found during ingestion and at least one elevation flag is set: asset_business_critical = {{ finding.asset_business_critical }}, asset_owner_privileged = {{ finding.asset_owner_privileged }}. Either flag alone is enough to unlock the critical impact matrix, allowing the rating to reach Critical.
{% else %}
Why complete (not elevated)? A CMDB record was found but neither elevation flag is set: asset_business_critical = False, asset_owner_privileged = {{ finding.asset_owner_privileged if finding.asset_owner_privileged is not none else "None" }}. The standard matrix applies — same outcome as the preliminary path, but the rating is now confirmed rather than estimated.
{% endif %}

Fields read: {{ bi_inputs }}

{# ── Tree 3: Urgency ──────────────────────────────────────────────────── #}

Tree 3 — Urgency → {{ explanation.urgency }}

Asks: how quickly must this finding be acted on? Takes the output of Tree 1 (not raw finding fields) as its first input. A High likelihood immediately gives Immediate regardless of anything else. Below High, patch availability and how many days the finding has been open versus the remediation threshold decide the outcome.

    {% for step in explanation.evidence_pointers.urgency.path %}
  1. {{ step }}
    2. {% endfor %}
{% if explanation.urgency == "Immediate" and urg_inputs.exploitation_likelihood == "High" %}
Why Immediate? Exploitation likelihood is High. This shortcuts every remaining check: patch availability and days open are not read at all. When a vulnerability is being actively exploited there is no safe waiting period regardless of remediation timelines.
{% elif explanation.urgency == "Immediate" %}
Why Immediate? Exploitation likelihood is not High, but the asset is internet-facing and the finding has exceeded its remediation threshold. Exposure plus an overdue SLA together justify the highest urgency even without confirmed exploitability data.
{% elif explanation.urgency == "Scheduled" %}
Why Scheduled? The conditions for Immediate were not met — likelihood is not High and the finding is either within its SLA window or no patch is yet available. The finding is real and should enter the next planned remediation cycle, but does not require emergency action today.
{% elif explanation.urgency == "Monitor" %}
Why Monitor? No exploit exists, no active exploitation is reported, and the finding is either within its SLA window or has no patch to apply. There is nothing actionable right now; re-evaluate at the next scheduled review.
{% endif %}

Fields read: {{ urg_inputs }}

Stage 3 — Explanation

The explanation layer converts the three rated outcomes into human language. No new decisions are made — every sentence derives from values already produced by abstraction. The resulting ExplanationObject is then passed unchanged to both stakeholder views.

Certainty

certainty_level
{{ explanation.certainty_level }}
{% if explanation.certainty_level == "Known" %}
Every field the three trees read was present and non-null in the NormalisedFinding. The ratings are based on complete information for this source format.
{% elif explanation.certainty_level == "Known uncertainty" %}
Some fields the trees needed were missing — but each missing field is listed in absent_by_format for {{ finding.source_format }}. The gaps are a known structural property of this source format, not a data quality problem. The rating is reliable given what the source can provide.
{% elif explanation.certainty_level == "Unknown uncertainty" %}
A field that this source format normally provides is unexpectedly null. The rating may be less reliable than it appears; investigate whether the source data is complete.
{% elif explanation.certainty_level == "Unknowable" %}
No CVE identifier is associated with this finding. Exploitability is undefined in principle for end-of-life or misconfiguration detections, so the exploitation likelihood tree cannot produce a meaningful rating.
{% endif %}

{# ─────────────────────────────────────────────────────────────────────────── STAGE 4 — VIEWS ─────────────────────────────────────────────────────────────────────────── #}

Stage 4 — Two renderings of the same object

The ExplanationObject built above is passed to two separate renderers. Neither renderer makes any decisions or adds any information — they each select a subset of the same object and format it for a different audience.

  • Technical view — shows the tree paths and inputs used, CVSS score, severity band, and all intermediate values. Intended for security analysts.
  • Non-technical view — shows only the risk summary, business consequence, time horizon, and action. No field names or scores. Intended for delivery leads and client-facing roles.

Adding a third audience (for example an executive one-pager) requires only one new renderer function and one new template — no other layer changes.
